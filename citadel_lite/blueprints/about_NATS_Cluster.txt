## 1. Architecture Overview

NATS serves as the **real-time event backbone** for the entire CNWB platform. It connects:

- **VPS services** (direct TCP) — Zayara bot, Nemesis daemon, voice engine, repair system
- **ECS Fargate workers** (direct TCP via service discovery) — CPU, data processing, notification tasks
- **Browser clients** (HTTP bridge → SSE/WebSocket) — SmartBank live updates, workshop sync
- **n8n automation** (HTTP bridge) — workflow triggers, SmartBank/CodeBank event routing
- **CI/CD pipeline** (direct TCP from GitLab Runner) — AI review results, SmartBank scans
- **Citadel Helper CLI** (HTTP bridge) — health probes, manual event publishing

### Design Principles

| Principle | Implementation |
|-----------|---------------|
| **Low latency** | NATS core PubSub for real-time events (<5ms on VPS) |
| **Durability** | JetStream for REFLEX events (7-day retention) |
| **Browser access** | HTTP bridge with SSE streaming + WebSocket fallback |
| **Offline resilience** | Client-side message queuing with auto-reconnect |
| **Audit trail** | All messages logged to `nats_message_log` (Supabase) |
| **Schema versioning** | `_meta.version` field on all browser-originated messages |

## 7. Client Implementations

### 7.1 Browser Client — `NATSService` (JavaScript)

**File:** `src/lib/natsService.js` (488 lines)
**SRS:** CNWB-NATS-SVC-001

The primary browser client. Supports HTTP bridge (SSE for subscribe, POST for publish) and WebSocket direct connection.

#### Class Structure

```javascript
class NATSService {
  constructor() {
    this.connected = false;
    this.userId = null;
    this.eventSource = null;           // SSE connection
    this.ws = null;                     // WebSocket connection
    this.subscriptions = new Map();     // subject → Set<callback>
    this.pendingMessages = [];          // Offline queue
    this.reconnectAttempts = 0;
    this.maxReconnectAttempts = 10;
    this.reconnectDelay = 1000;         // Exponential backoff base
  }
}
```

#### Key Methods

| Method | Purpose | Subject |
|--------|---------|---------|
| `connect(userId)` | Establish SSE/WS connection | — |
| `disconnect()` | Close connection, drain queue | — |
| `publish(subject, payload, options)` | POST to bridge or WS send | Any |
| `subscribe(subject, callback)` | Register local handler | Any (wildcard support) |
| `unsubscribe(subject, callback)` | Remove local handler | Any |
| `publishXPUpdate(userId, data)` | XP balance event | `citadel.smartbank.{userId}.xp` |
| `publishTPUpdate(userId, data)` | TP balance event | `citadel.smartbank.{userId}.tp` |
| `publishRankChange(userId, data)` | Rank progression | `citadel.smartbank.{userId}.rank` |
| `publishCodeBankEvent(userId, data)` | Code analysis | `citadel.codebank.{userId}.events` |
| `publishWorkshopSync(userId, data)` | Session sync | `citadel.workshop.{userId}.sync` |
| `publishBookmakerEvent(userId, topic, data)` | Book generation | `citadel.bookmaker.{userId}.{topic}` |
| `sendAgentCommand(agentId, command)` | A2A command | `citadel.a2a.{agentId}.command` |

#### React Hook

```javascript
export function useNATS(userId) {
  // Auto-connects on mount, disconnects on unmount
  // Returns: { nats, connected, publish, subscribe }
  const [connected, setConnected] = useState(false);
  const natsRef = useRef(natsService);

  useEffect(() => {
    if (userId) natsRef.current.connect(userId);
    return () => natsRef.current.disconnect();
  }, [userId]);

  return {
    nats: natsRef.current,
    connected,
    publish: natsRef.current.publish.bind(natsRef.current),
    subscribe: natsRef.current.subscribe.bind(natsRef.current),
  };
}
```

#### Wildcard Matching

The browser client implements NATS-compatible wildcard matching locally:

| Pattern | Matches | Example |
|---------|---------|---------|
| `*` | Single token | `citadel.*.ping` matches `citadel.smartbank.ping` |
| `>` | Multi token (tail) | `citadel.repair.>` matches `citadel.repair.pipeline.trigger` |
| Exact | Exact match | `tasks.cpu` matches `tasks.cpu` only |

#### Offline Queuing

When disconnected, messages are stored in `pendingMessages[]` and automatically flushed upon reconnection. Reconnection uses exponential backoff (1s, 2s, 4s, ... up to 10 attempts).

### 7.2 ECS Fargate Worker (Python)

**File:** `worker/nats_worker.py` (197 lines)
**SRS:** SRS-NATS-WORKER-001

Consumes tasks from NATS and processes them on ECS Fargate Spot instances.

#### Handler Registry

```python
HANDLERS = {}

def register_handler(category):
    """Decorator to register task handlers by category."""
    def decorator(func):
        HANDLERS[category] = func
        return func
    return decorator

@register_handler("cpu")
async def handle_cpu_task(payload):
    # CPU-intensive processing
    ...

@register_handler("data_processing")
async def handle_data_processing(payload):
    # ETL / transformation
    ...

@register_handler("notification")
async def handle_notification(payload):
    # Notification dispatch
    ...
```

#### Lifecycle

1. Connect to `NATS_URL`
2. Subscribe to `tasks.{category}` for each category in `WORKER_CATEGORIES`
3. On message: decode JSON, look up handler, execute, publish result to `results.{task_id}`
4. Heartbeat every 30s to `workers.heartbeat`
5. On SIGTERM/SIGINT: drain connection (finish in-flight), exit

#### Docker

```dockerfile
FROM python:3.11-slim
WORKDIR /app
RUN pip install --no-cache-dir nats-py psutil
COPY nats_worker.py .
ENV NATS_URL=nats://localhost:4222
ENV WORKER_CATEGORIES=cpu,data_processing,notification
CMD ["python", "-m", "nats_worker"]
```

### 7.3 Task Distributor (Python)

**File:** `server/services/task_distributor.py` (146 lines)
**SRS:** SRS-TASKDIST-001

VPS-side service that distributes tasks to ECS workers via NATS request/reply.

```python
class TaskDistributor:
    async def connect(self):
        self._nc = await nats_lib.connect(self.nats_url)
        await self._nc.subscribe("results.*", cb=self._handle_result)

    async def submit_task(self, category, payload, timeout=30):
        """Submit task and wait for result (request/reply pattern)."""
        task_id = str(uuid.uuid4())
        future = asyncio.Future()
        self._pending[task_id] = future
        msg = json.dumps({
            "task_id": task_id,
            "category": category,
            "payload": payload,
            "reply_subject": f"results.{task_id}"
        })
        await self._nc.publish(f"tasks.{category}", msg.encode())
        return await asyncio.wait_for(future, timeout=timeout)

    async def publish_event(self, subject, payload):
        """Fire-and-forget event publishing."""
        await self._nc.publish(subject, json.dumps(payload).encode())
```

Singleton access via `get_task_distributor()`.

Request/reply pattern — publish and wait for a response.

**Request:**

**Response:** The reply message payload.

## 9. WebSocket & SSE API

### WebSocket (`wss://nats.citadel-nexus.com`)

Direct NATS.ws connection for browsers. Lower latency than HTTP bridge but requires WebSocket support.

**Connection:**
```javascript
const ws = new WebSocket('wss://nats.citadel-nexus.com');
ws.onopen = () => {
  // Send NATS CONNECT protocol
  ws.send(JSON.stringify({ /* NATS CONNECT */ }));
};
```

**Used when:** `VITE_NATS_WS_URL` is configured and WebSocket connection succeeds. Falls back to SSE if unavailable.

### SSE (`/subscribe?userId={id}`)

Server-Sent Events stream for receiving NATS messages. One-directional (receive only). Publishing still uses HTTP POST.

**Connection:**
```javascript
const es = new EventSource(
  `${NATS_CONFIG.httpBridgeUrl}/subscribe?userId=${userId}`
);
es.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  this.handleIncomingMessage(msg);
};
```

**Reconnection:** Browser `EventSource` auto-reconnects. Custom reconnection logic in `NATSService` with exponential backoff.

## 12. ECS Worker System

### Architecture

```
VPS CPU > 80%
  → CloudWatch Alarm (vps-cpu-high)
    → ECS Auto-Scaling Policy
      → Scale worker-service desired_count +1
        → Fargate Spot Task launches
          → nats_worker.py connects to NATS
            → Subscribes to tasks.{category}
            → Processes tasks, publishes results
            → Heartbeats every 30s

VPS CPU < 40%
  → CloudWatch Alarm (vps-cpu-low)
    → Scale worker-service desired_count -1
      → Worker drains connection, exits
```

### Terraform Resources

**Source:** `infra/terraform/aws/scaling.tf`

| Resource | Purpose |
|----------|---------|
| `aws_ecs_task_definition.worker` | Worker container definition (NATS_URL, WORKER_CATEGORIES) |
| `aws_ecs_service.worker` | Service with `desired_count = 0`, Fargate Spot |
| `aws_appautoscaling_target.worker` | Min 0, Max 3 tasks |
| `aws_cloudwatch_metric_alarm.vps_cpu_high` | Scale up at CPU > 80% |
| `aws_cloudwatch_metric_alarm.vps_cpu_low` | Scale down at CPU < 40% |

### Worker Categories

| Category | Handler | Use Case |

## 18. Security & Authentication

### Current Security Model

| Layer | Mechanism | Status |
|-------|-----------|--------|
| **Transport (TCP)** | Token auth via `NATS_TOKEN` | Optional — used when configured |
| **Transport (HTTP Bridge)** | nginx TLS termination | Active — HTTPS enforced |
| **Transport (WebSocket)** | WSS via nginx | Active — TLS enforced |
| **Authorization** | No per-subject ACLs | Not yet implemented |
| **Message integrity** | No signing | Not yet implemented |
| **Audit** | `nats_message_log` table | Active — all messages logged |

### Recommendations

1. **Enable NATS token auth** on all direct TCP connections (VPS services, ECS workers)
2. **Add per-subject ACLs** — workers should only subscribe to `tasks.*`, not `citadel.*`
3. **HTTP bridge auth** — add API key or JWT validation for `/publish` endpoint
4. **Rate limiting** — prevent flood attacks on bridge publish endpoint
5. **Message signing** — add HMAC on sensitive economy messages (XP/TP)

## 20. Failure Modes & Recovery

### Connection Failures

| Failure | Impact | Recovery |
|---------|--------|----------|
| **NATS server down** | All real-time features offline | Systemd auto-restart; n8n webhooks buffer |
| **HTTP bridge down** | Browser, n8n, Helper can't publish | nginx 502 → clients retry; SSE auto-reconnect |
| **WebSocket disconnected** | Browser falls back to SSE | `NATSService` exponential backoff (10 attempts) |
| **ECS worker disconnected** | Tasks not processed | `drain()` → clean exit → ECS replaces task |
| **n8n workflow fails** | Events not routed | n8n retry logic (configurable per node) |

### Message Loss Scenarios

| Scenario | Prevention |
|----------|-----------|
| **Core PubSub** (no JetStream) | Messages lost if no subscriber is active. Use JetStream for critical subjects. |
| **Browser offline** | `pendingMessages[]` queue; flushed on reconnect |
| **Worker crash mid-task** | No result published → `submit_task()` timeout → caller retries |
| **HTTP bridge timeout** | `httpx` 10s timeout → exception → caller decides retry |

### Recovery Procedures

1. **NATS server restart:** `sudo systemctl restart nats-server`
2. **HTTP bridge restart:** `sudo systemctl restart nats-bridge`
3. **Verify connectivity:** `nats pub test "hello"` from VPS
4. **Check JetStream:** `nats stream ls`, `nats consumer ls REFLEX_EVENTS`
5. **Verify workers:** Check `workers.heartbeat` subject for active workers
6. **Replay messages:** Query `nats_message_log` and republish via HTTP bridge
