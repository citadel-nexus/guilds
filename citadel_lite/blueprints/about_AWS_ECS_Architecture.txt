### Design Principles

| Principle | Implementation |
|-----------|---------------|
| **Cost-first** | $400/mo hard cap; VPS as primary; ECS only on overflow |
| **Federated processing** | NATS distributes light tasks to Fargate; Ray handles heavy compute |
| **Zero-downtime migration** | nginx upstream switch + health checks before cutover |
| **Budget-aware AI** | LLM + rule fallback decides migration based on CPU, memory, cost |
| **Spot-first** | All ECS workers and Ray nodes use Spot instances |
| **One-at-a-time** | Only one service migrates at a time to limit blast radius |

### How Tasks Are Distributed

1. **VPS handles** (always-on): Real-time requests, Redis operations, NATS server, database queries, nginx routing
2. **NATS Workers handle** (Fargate Spot, scale 0вҖ“3): CPU-intensive processing, data transformation, notification dispatch
3. **Ray Workers handle** (EC2 Spot, scale 0вҖ“2): Embedding generation, model inference, large ETL jobs

### Distribution Decision Flow

```python
# Compute Bridge (server/services/compute_bridge.py)
if task.category in ("cpu", "data_processing", "notification"):
    # Light task вҶ’ NATS вҶ’ Fargate Spot worker
    result = await task_distributor.submit_task(task.category, task.payload)
else:
    # Heavy task вҶ’ Ray вҶ’ EC2 Spot worker
    result = await ray_client.submit(task)
```

### NATS-Based Task Federation

```
VPS Task Distributor
  в”Ӯ
  в”ңв”Җв”Җ publish("tasks.cpu", payload)        вҶ’ Fargate Worker #1 subscribes
  в”ңв”Җв”Җ publish("tasks.data_processing", p)  вҶ’ Fargate Worker #2 subscribes
  в”ңв”Җв”Җ publish("tasks.notification", p)     вҶ’ Fargate Worker #3 subscribes
  в”Ӯ
  в””в”Җв”Җ subscribe("results.*")              вҶҗ Workers publish results back
```

Each worker:
- Subscribes to `tasks.{category}` for its configured categories
- Processes tasks via registered handler functions
- Publishes results to `results.{task_id}`
- Sends heartbeat every 30s to `workers.heartbeat`

### Parallel Processing Guarantees

| Property | Mechanism |
|----------|-----------|
| **At-most-once delivery** | NATS core PubSub (no JetStream for tasks) |
| **Result correlation** | `task_id` in request вҶ’ `results.{task_id}` subject |
| **Timeout** | Configurable per `submit_task()` call (default 30s) |
| **Worker failure** | No heartbeat вҶ’ CloudWatch alarm вҶ’ ECS replaces task |
| **Graceful shutdown** | SIGTERM вҶ’ `nc.drain()` вҶ’ finish in-flight вҶ’ exit |

### Migration Thresholds

| Threshold | Value | Action |
|-----------|-------|--------|
| VPS CPU > 80% | `_MIGRATE_CPU_THRESHOLD` | Migrate highest-CPU container to ECS |
| Container CPU > 15% | `_CONTAINER_CPU_THRESHOLD` | Eligible for migration |
| VPS CPU < 40% | `_REVERSE_CPU_THRESHOLD` | Migrate back to VPS (cost savings) |
| Budget > 80% | `_BUDGET_WARNING_PCT` | Start reversing lowest-priority services |
| Budget > 90% | `_BUDGET_CRITICAL_PCT` | Force all services back to VPS |
| Concurrent migrations | 1 | Only one service migrates at a time |

### Migration Decision Priority

1. **Budget critical** (вүҘ90%) вҶ’ Force ALL back to VPS
2. **Budget warning** (вүҘ80%) вҶ’ Reverse one low-priority service
3. **VPS CPU low** (<40%) вҶ’ Reverse one service (cost savings)
4. **VPS CPU high** (вүҘ80%) вҶ’ Migrate one high-CPU container to ECS

### Migration Execution Steps

#### VPS вҶ’ ECS

```bash
# 1. Scale ECS service to 1
aws ecs update-service --cluster citadel-cluster --service <svc>-service --desired-count 1

# 2. Wait for ECS task health check (60s start period)
# ... CloudWatch health check passes ...

# 3. Switch nginx upstream to ECS ALB
python3 nginx_route_manager.py switch <service> ecs --alb-dns <alb-dns>

# 4. Stop VPS Docker container
docker compose -f docker-compose.runtime.yaml stop <service>

# 5. Verify traffic flows through ECS (no error rate increase)
```

#### ECS вҶ’ VPS

```bash
# 1. Start VPS Docker container
docker compose -f docker-compose.runtime.yaml start <service>

# 2. Wait for VPS health check (30 attempts Г— 2s = 60s max)
curl -sf http://localhost:<port>/health

# 3. Switch nginx upstream back to VPS
python3 nginx_route_manager.py switch <service> vps

# 4. Scale ECS service to 0
aws ecs update-service --cluster citadel-cluster --service <svc>-service --desired-count 0
```

### LLM vs Rule Mode

The MigrateOrchestrator supports two decision modes:

| Mode | Engine | When Used |
|------|--------|-----------|
| **LLM mode** | Bedrock (Claude Haiku/Sonnet/Opus) | When LLM client is available and responsive |
| **Rule mode** | Threshold-based deterministic logic | Fallback when LLM unavailable |

LLM mode provides nuanced decisions considering cross-service dependencies and risk estimates. Rule mode uses the fixed threshold priorities above.

## 19. Deployment & CI/CD Integration

### GitLab CI вҶ’ ECS Deploy

The `.gitlab-ci.yml` pipeline includes jobs that:
1. Build Docker images on `cnwb-build` runner
2. Push to ECR repositories
3. Force new deployment via `aws ecs update-service --force-new-deployment`

### AWS Agent CLI

```bash
# Check ECS cluster status
python -m citadel_lite.src.agents.aws_agent status

# Scale a service
python -m citadel_lite.src.agents.aws_agent scale --service worker --count 2

# Force new deployment
python -m citadel_lite.src.agents.aws_agent deploy --service workshop

# Comprehensive health check (7-point)
python -m citadel_lite.src.agents.aws_agent health-check

# Get VPS metrics
python -m citadel_lite.src.agents.aws_agent vps-metrics --minutes 10

# Get per-container metrics
python -m citadel_lite.src.agents.aws_agent container-metrics --minutes 10

# View CloudWatch alarms
python -m citadel_lite.src.agents.aws_agent alarms

# S3 bucket stats
python -m citadel_lite.src.agents.aws_agent s3-stats
```

### AWS Agent A2A Protocol

The AWS agent also registers with the Agent-to-Agent (A2A) protocol:

```python
register_aws_agent(protocol)
# Capabilities: ecs-control, bedrock-invoke, s3-storage,
#   cloudwatch-logs, cloudwatch-metrics, auto-scaling,
#   s3-lifecycle, health-check
```

### Health Check (7-Point)

The AWS agent performs comprehensive infrastructure health checks:

1. **ECS Services** вҖ” desired vs running counts
2. **CloudWatch Alarms** вҖ” any in ALARM state
3. **CloudWatch VPS Metrics** вҖ” data freshness
4. **S3 Bucket** вҖ” accessibility and size
5. **Auto-scaling** вҖ” policies configured
6. **ECR Repositories** вҖ” accessible
7. **Budget** вҖ” utilization percentage

Returns: `healthy`, `degraded`, or `critical` status.
